# Nano-vLLM 复现学习计划

> 目标：通过逐步实现每一个模块，最终从零复现 Nano-vLLM —— 一个轻量级 vLLM 实现，理解现代 LLM 推理引擎的核心机制（KV Cache、PagedAttention、Prefix Cache、Tensor Parallelism、CUDA Graph 等）。

---

## 项目总览

Nano-vLLM 是一个用 ~1200 行 Python 实现的 vLLM 简化版，支持 Qwen3 模型推理。整体架构分为 4 大层：

```
┌─────────────────────────────────────────────────┐
│  对外 API：LLM / SamplingParams (用户接口)        │
├─────────────────────────────────────────────────┤
│  引擎层 engine：LLMEngine + Scheduler            │
│   + BlockManager + Sequence + ModelRunner       │
├─────────────────────────────────────────────────┤
│  模型层 models：Qwen3ForCausalLM                  │
├─────────────────────────────────────────────────┤
│  算子层 layers：Linear / Attention / RMSNorm /    │
│   RoPE / Embedding / Activation / Sampler        │
└─────────────────────────────────────────────────┘
            ↑ utils: context（全局上下文）+ loader（权重加载）
```

核心文件清单（参考阅读，不要在学习前直接抄）：
- 配置：[`config.py`](nanovllm/config.py)、[`sampling_params.py`](nanovllm/sampling_params.py)
- 引擎：[`engine/sequence.py`](nanovllm/engine/sequence.py)、[`engine/block_manager.py`](nanovllm/engine/block_manager.py)、[`engine/scheduler.py`](nanovllm/engine/scheduler.py)、[`engine/model_runner.py`](nanovllm/engine/model_runner.py)、[`engine/llm_engine.py`](nanovllm/engine/llm_engine.py)
- 模型：[`models/qwen3.py`](nanovllm/models/qwen3.py)
- 算子：[`layers/`](nanovllm/layers) 下 7 个文件
- 工具：[`utils/context.py`](nanovllm/utils/context.py)、[`utils/loader.py`](nanovllm/utils/loader.py)

- [ ] 2.4 实现 [`layers/linear.py`](nanovllm/layers/linear.py) **单卡部分**：

### 阶段 3：上下文与 KV Cache 算子（1 天）

**目标**：实现推理时的全局 Context，以及把 K/V 写入 KV Cache 的 Triton 算子。

- [ ] 3.1 实现 [`utils/context.py`](nanovllm/utils/context.py)：

- [ ] 3.2 实现 [`layers/attention.py`](nanovllm/layers/attention.py) 的 Triton 算子 `store_kvcache_kernel`：

- [ ] 3.3 实现 `Attention.forward`：

  - 在 decode 路径上调 `flash_attn_with_kvcache`
  - K/V 同时写入 cache

**验收**：构造一个小批 prefill + decode 流程，与 HuggingFace `Qwen3ForCausalLM` 输出对齐。

### 阶段 4：组装 Qwen3 模型 + 权重加载（1 天）

**目标**：完整还原 Qwen3 模型结构，并能从 safetensors 正确加载权重。

- [ ] 4.1 实现 [`models/qwen3.py`](nanovllm/models/qwen3.py) 中各模块：
  - `Qwen3Attention`：QKV 合并投影、QK RMSNorm、RoPE、Attention、O 投影
  - `Qwen3MLP`：gate_up 合并 + SiluAndMul + down_proj
  - `Qwen3DecoderLayer`：注意 residual 在 input_layernorm 内被融合
  - `Qwen3Model` / `Qwen3ForCausalLM`：注意 `tie_word_embeddings` 时 lm_head 共享 embed_tokens 权重
  - 类属性 `packed_modules_mapping`：把 `q_proj/k_proj/v_proj` 映射到合并后的 `qkv_proj`
- [ ] 4.2 实现 [`utils/loader.py`](nanovllm/utils/loader.py)：
  - 遍历目录里的 `*.safetensors`
  - 对每个 weight_name，先查 `packed_modules_mapping` 是否需要分片合并加载
  - 通过 `param.weight_loader(...)` 调用各 Parameter 自带的加载器

**验收**：加载 Qwen3-0.6B 的真实权重，做一次单 batch、单步 forward，与 HuggingFace 推理结果 logits 比对一致。

### 阶段 5：BlockManager —— 分页 KV Cache 与 Prefix Cache（1 天）

**目标**：理解并实现 PagedAttention 的「内存分页 + 哈希前缀复用」。

- [ ] 5.1 实现 [`engine/block_manager.py`](nanovllm/engine/block_manager.py) 的 `Block` 类：`block_id / ref_count / hash / token_ids`
- [ ] 5.2 实现 `BlockManager`：
  - `free_block_ids`(deque) + `used_block_ids`(set) + `hash_to_block_id`(dict)
  - `compute_hash`：用 `xxhash.xxh64` 链式哈希（带 prefix）
  - `_allocate_block / _deallocate_block`：处理 hash 字典清理
  - `can_allocate(seq) -> num_cached_blocks` ：扫描 prompt 块查命中前缀缓存，返回可复用块数
  - `allocate(seq, num_cached_blocks)` ：先复用命中块（ref_count++），再分配剩余新块
  - `deallocate(seq)`：逆序减引用计数，归还块
  - `can_append / may_append`：decode 时只在 token 跨过 block 边界才申请新块
  - `hash_blocks(seq)`：prefill/decode 写完一个 block 后立即计算 hash 并登记
- [ ] 5.3 用单元测试验证：相同 prompt 第二次进来时，`num_cached_blocks` 等于 `num_blocks-1`（最后一个不完整块不哈希）

**验收**：能解释 ref_count、hash 链、为什么最后一个块不参与哈希。

### 阶段 6：Scheduler —— 调度策略（0.5 天）

**目标**：实现 vLLM 风格的 Continuous Batching + Chunked Prefill + 抢占式调度。

- [ ] 6.1 实现 [`engine/scheduler.py`](nanovllm/engine/scheduler.py)：
  - 双队列 `waiting / running`
  - `schedule()` 优先尝试 prefill（受 `max_num_batched_tokens / max_num_seqs` 限制）
    - 第一个序列允许 chunked prefill（剩余 token 数 < 序列长度）
    - 满足整段 prefill 后状态切到 RUNNING，移到 running 队列
  - 没有 prefill 任务再做 decode：每个 running 序列 schedule 1 token；空间不足时**从队尾抢占**（preempt）压回 waiting
  - `preempt(seq)`：状态回 WAITING、`is_prefill=True`、释放块、放回 waiting 队首
  - `postprocess(seqs, token_ids, is_prefill)`：写入 hash、累加缓存计数、追加新 token、检查终止条件（EOS / max_tokens）

**验收**：构造 100+ 序列的小压测，观察 prefill/decode 切换、抢占触发不报错。

### 阶段 7：ModelRunner（单卡基础版）（1 天）

**目标**：把 schedule 的输出转化为模型输入张量，跑模型，再把 logits 采样输出。先做 **单卡 + enforce_eager** 简化版。

- [ ] 7.1 实现 [`engine/model_runner.py`](nanovllm/engine/model_runner.py) 单卡部分：
  - `__init__`：单卡时 `dist.init_process_group` 也要初始化（用 size=1），创建模型、加载权重、warmup、分配 KV Cache
  - `warmup_model`：构造一个最大 batch 的假 prefill 跑一次，预热 + 触发 torch.compile
  - `allocate_kv_cache`：根据 `gpu_memory_utilization` 计算可用块数，分配大张量 `(2, num_layers, num_blocks, block_size, num_kv_heads, head_dim)`，按 layer 把切片塞回 `Attention.k_cache/v_cache`
  - `prepare_prefill(seqs)`：拼 input_ids、positions、cu_seqlens、slot_mapping，处理「已缓存前缀」情形
  - `prepare_decode(seqs)`：每序列 1 token，slot_mapping 用 `block_table[-1] * block_size + last_block_num_tokens - 1`
  - `prepare_block_tables`：padding 到 max_len
  - `prepare_sample`：温度
  - `run_model` / `run` / `call`：先只做 eager 路径

**验收**：能在单卡上跑通 [`example.py`](example.py) 输出和 HuggingFace 一致或合理。

### 阶段 9：Tensor Parallelism（1 天）

**目标**：在已通过单卡测试的基础上，加上多卡并行。

- [ ] 9.1 升级 [`layers/linear.py`](nanovllm/layers/linear.py) 为完整 TP 版本：
  - `ColumnParallelLinear`：output 维度切分；`weight_loader` 按 `tp_rank` 切片
  - `RowParallelLinear`：input 维度切分；forward 末尾 `dist.all_reduce`
  - `QKVParallelLinear`：按 `q/k/v` 三段加载，每段在 tp 维度切片
  - `MergedColumnParallelLinear`：按 list 偏移加载
- [ ] 9.2 升级 [`layers/embed_head.py`](nanovllm/layers/embed_head.py) 的 TP：
  - VocabParallelEmbedding：按 vocab 维切；forward 时屏蔽不在本 rank 的 id，最后 all_reduce
  - ParallelLMHead：rank0 上 gather 全 vocab logits
- [ ] 9.3 升级 [`engine/model_runner.py`](nanovllm/engine/model_runner.py)：
  - 用 `mp.spawn` 启动 worker 进程
  - 共享内存（SharedMemory + Event）做主控 → worker 的命令广播
  - rank!=0 进入 `loop()` 等待命令；rank0 通过 `call/write_shm` 触发
  - 仅 rank0 做采样并返回 token_ids

**验收**：`tensor_parallel_size=2`（如果有多卡）能跑出正确结果，吞吐有提升。

### 阶段 10：CUDA Graph 优化 decode（0.5 天）

**目标**：把 decode 路径用 CUDA Graph 捕获多个固定 batch_size 的图，replay 时只搬运指针，绕开 Python overhead。

- [ ] 10.1 在 ModelRunner 实现 `capture_cudagraph`：
  - 选定 `graph_bs = [1,2,4,8,16,32,...,max_bs]`
  - 从大到小捕获，每个 bs 一张 graph，共享 `graph_pool`
  - 预先分配「输入/输出大张量」放在 `graph_vars`
- [ ] 10.2 在 `run_model` 中：非 prefill 且非 enforce_eager 且 batch ≤ 512 时，
  - 选择 ≥ bs 的最小图 → 拷输入数据 → `graph.replay()` → 切片输出
  - 注意 `slot_mapping` padding 部分要填 `-1`、`context_lens` 清零
- [ ] 10.3 测试：`enforce_eager=False` 跑 [`bench.py`](bench.py) 对比开/关 graph 的吞吐

**验收**：decode 阶段有可见加速。

## 学习建议

1. **不要一开始抄所有文件**：每个阶段做完写一份「我学到了什么」的笔记，再去对照原码看差异。
2. **多打 print + 单步验证**：尤其是 BlockManager / Scheduler，状态机复杂，多日志能快速建立直觉。
3. **优先理解再优化**：torch.compile / CUDA Graph / Triton kernel 是优化项，理解逻辑比性能更重要。
4. **善用对照实现**：`transformers.Qwen3ForCausalLM` 是天然的 reference，可以拿它的中间结果对每一层算子做 diff。
5. **遇到 flash-attn 不可用**：可先用 `torch.nn.functional.scaled_dot_product_attention` 实现一个慢版本，跑通逻辑后再换 flash-attn。
